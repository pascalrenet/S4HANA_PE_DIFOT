1. What is DIFOT?

DIFOT stands for Delivered In Full, On Time. It is a supply chain performance metric that measures whether a supplier delivered exactly what was ordered and when it was promised.

A delivery only achieves a DIFOT score if it satisfies both conditions simultaneously:

📦
In Full
Qty GR ≥ Ordered
📅
On Time
GR Date ≤ Sched. Date
✅
Result: DIFOT
Both met
❌
Result: NOT DIFOT
Either or both failed

A third status, PENDING, applies when no goods receipt has been posted yet — the PO item exists and has a scheduled delivery date, but the supplier has not yet delivered. This is equally important: it tells procurement exactly which open orders are overdue or at risk.

When a delivery fails DIFOT, the report also assigns a Failure Reason:

Failure ReasonMeaning
SHORTQuantity received was less than ordered. The delivery date was met but stock is short.
LATEFull quantity was delivered but after the scheduled date.
SHORT AND LATEBoth quantity and date failed simultaneously.
blankDelivery achieved DIFOT — no failure to report.

At each purchase order item, SAP stores a set of schedule lines — each with a delivery date and an expected quantity. When goods arrive at the warehouse and a goods receipt is posted, SAP records the actual quantity and posting date against the PO. DIFOT compares these two sides: what was promised versus what was actually received.

DIFOT sits at the intersection of procurement, warehouse operations, and supplier management. It is commonly used to:

  • Score and rank supplier performance for periodic business reviews
  • Identify systemic delivery problems with specific suppliers, materials, or plants
  • Provide objective evidence when renegotiating contracts or applying penalty clauses
  • Feed into broader supply chain KPIs and executive dashboards
  • Detect trends — for example, a supplier whose on-time rate is declining over consecutive months

Industry context: DIFOT (also written as OTIF — On Time In Full — in some industries) is a standard KPI in retail, FMCG, automotive, and manufacturing supply chains. Retailers such as supermarket chains often impose financial penalties on suppliers whose DIFOT rates fall below a contractual threshold (commonly 95–98%).

3. The Problem This Project Solves

Despite DIFOT being a critical KPI, most SAP customers have historically struggled to get a clean, real-time view of it. The typical pain points are:

No standard SAP report

SAP S/4HANA does not ship a dedicated DIFOT report. Procurement teams typically cobble together answers from the Purchase Order Monitor, GR/GI Slip reports, and manual Excel spreadsheets — a process that is error-prone, slow, and inconsistent across users.

Data is spread across multiple tables

The data needed to score DIFOT lives in at least three separate areas of the SAP data model:

  • PO header and item data (ordered quantity, supplier, delivery date) — EKKO / EKPO
  • Schedule line dates — EKET
  • Goods receipt history — EKBE

Joining and aggregating these correctly — especially netting GR reversals, handling split deliveries, and dealing with items that have no GR yet — requires careful logic that is easy to get wrong in ad-hoc reporting.

No drill-down from summary to detail

Even when a summary view exists, procurement staff need to drill into individual goods receipt movements to investigate discrepancies. A static report or flat extract does not support this.

What this project delivers

This project delivers a real-time, embedded Fiori List Report directly on the S/4HANA Cloud Fiori Launchpad. It reads live transactional data — no extracts, no overnight jobs, no spreadsheets. Procurement staff can filter by supplier, material, plant, DIFOT status, or failure reason, drill into any PO item to see all four DIFOT dimensions on an Object Page, and further drill into individual goods receipt lines to investigate the root cause of a failure.

4. The Tools We Used

SAP S/4HANA Cloud Public Edition

The target system. All custom CDS views, service definitions, and the deployed Fiori app live here. We used the Developer Extensibility model, which allows custom ABAP Cloud development on top of released SAP APIs.

ABAP Development Tools (ADT)

Eclipse-based IDE for creating and activating CDS Data Definitions, Service Definitions, Service Bindings, IAM Apps, and Business Catalogs in the S/4HANA Cloud back-end.

SAP Business Application Studio (BAS)

SAP's cloud-based IDE (running on BTP) used to generate the Fiori Elements application from a template wizard, preview it against live data, and deploy it as a BSP application to the ABAP repository.

SAP BTP (Business Technology Platform)

Provides the BAS environment and the BTP destination that allows BAS to connect to and deploy into the S/4HANA Cloud system.

ABAP CDS (Core Data Services)

The primary development artefact. CDS views define the data model, join logic, aggregations, DIFOT calculations, and all Fiori UI annotations — no classical ABAP report or BAdI required.

OData V4

The service protocol. A Service Definition and Service Binding expose the CDS model as an OData V4 endpoint that BAS and the deployed Fiori app consume.

SAP Fiori Elements

The UI framework. Because all UI behaviour is driven by CDS annotations, the Fiori Elements List Report and Object Page are generated automatically — no custom JavaScript or UI5 coding.

Released SAP APIs

All data access uses released, stable I_* CDS views (I_PurchaseOrderItemAPI01, I_PurchaseOrderHistoryAPI01, etc.) — no direct table access, ensuring upgrade compatibility.

5. Solution Architecture

The solution is built in layers. Each layer has a single responsibility and feeds the next:

Layer responsibilities

LayerObject(s)Responsibility
Released APIsI_PurchaseOrderItemAPI01, I_PurchaseOrderHistoryAPI01, etc.Stable, upgrade-safe access to PO, GR, and schedule line data. No direct table access anywhere in this solution.
AggregationZC_PRPOItemGRSummary
ZC_PRPOSchedLineSummary		Aggregate GR quantities (netting reversals) and scheduled delivery dates per PO item. Isolate aggregation logic so it can be reused.
GR DetailZC_PRPOGRHistoryLinesIndividual (non-aggregated) GR lines — used exclusively as a navigation target for the Object Page drill-down table.
Core CalculationZC_PRPODIFOTJoin all inputs; calculate QuantityVariance, DateVarianceInDays, IsDeliveredInFull, IsDeliveredOnTime, DIFOTStatus, and DIFOTFailureReason.

6. How the Report Works

The List Report

When a user opens the app, they see a Fiori Elements List Report with a Smart Filter Bar at the top. The filter bar exposes the most useful search dimensions — Purchase Order, Supplier, Material, Plant, Material Group, DIFOT Status (dropdown), Failure Reason (dropdown), and Earliest Scheduled Delivery Date.

After clicking Go, the table populates with one row per PO item. The key columns are:

ColumnMeaning
Ordered QtyQuantity on the PO item
GR QuantityNet goods receipts posted (reversals already netted off)
Qty VarianceGR Qty minus Ordered Qty — negative means short delivery
Sched Del. DateLatest scheduled delivery date from the PO schedule lines
Latest GR DateDate the most recent goods receipt was posted
Date Var. (Days)Days between scheduled and actual — positive means late
In Full?Tick if quantity met
On Time?Tick if date met
DIFOT StatusDIFOT   NOT DIFOT   PENDING — colour-coded with icon
Failure ReasonSHORT / LATE / SHORT AND LATE / blank

The Object Page drill-down

Clicking any row opens an Object Page with four tabs:

  • PO Details — supplier, company code, purchasing org/group, material, plant, net price, etc.
  • DIFOT Details — the four DIFOT fields and failure reason, displayed with criticality colouring
  • Schedule & GR — scheduled quantities and dates, total GR quantity, first and latest GR dates, supplier confirmations
  • Goods Receipt Lines — every individual GR and reversal movement (material document, movement type, posting date, quantity, amount), making it easy to trace exactly what arrived and when

The DIFOT logic in brief

-- Delivered In Full:
IsDeliveredInFull = 'X'  when  TotalGRQuantity >= OrderQuantity

-- Delivered On Time:
IsDeliveredOnTime = 'X'  when  LatestGRPostingDate <= LatestSchedDelivDate

-- Overall Status:
DIFOTStatus = 'PENDING'   when no GR has been posted
DIFOTStatus = 'DIFOT'     when both InFull AND OnTime
DIFOTStatus = 'NOT DIFOT' otherwise

-- Failure Reason:
DIFOTFailureReason = 'SHORT AND LATE'  when short AND late
DIFOTFailureReason = 'SHORT'           when short only
DIFOTFailureReason = 'LATE'            when late only
DIFOTFailureReason = ''                when DIFOT or PENDING

Design decision — split deliveries: Where a PO item has multiple schedule lines, we compare against the latest scheduled delivery date. This avoids false DIFOT failures on split deliveries — a supplier who delivers the full quantity by the last scheduled line date is scored as on time.

7. Replicating This in Your System

Prerequisites: You need an S/4HANA Cloud Public Edition system with developer extensibility enabled, ADT (Eclipse) connected to your system, a SAP Fiori Dev Space in BAS, and a BTP destination pointing to your S/4HANA Cloud system. Your user needs business catalogs SAP_A4C_BC_DEV_UID_PC, SAP_CORE_BC_EXT_TST, and SAP_A4C_BC_DEV_PC assigned.

The implementation follows a strict activation order because each CDS view depends on the ones below it. The full sequence is:

Phase 1 — Build the CDS layer in ADT

  1. Create a custom package in ADT (e.g. ZPURCHASING). Assign it to a transport request. All subsequent objects go into this package.
  2. Create ZC_PRPOItemGRSummary — aggregates net GR quantities per PO item from I_PurchaseOrderHistoryAPI01. Filters on PurchasingHistoryCategory = 'E' and movement types 101/102; uses DebitCreditCode to net reversals. Produces TotalGRQuantity, FirstGRPostingDate, LatestGRPostingDate. 
    define view ZC_PRPOItemGRSummary
  as select from I_PurchaseOrderHistoryAPI01
{
  key PurchaseOrder,
  key PurchaseOrderItem,
  sum( case DebitCreditCode
         when 'S' then Quantity
         when 'H' then -Quantity
         else 0
       end )              as TotalGRQuantity,
  max( PostingDate )      as LatestGRPostingDate,
  min( case DebitCreditCode when 'S' then PostingDate end )
                          as FirstGRPostingDate
}
where PurchasingHistoryCategory = 'E'
  and ( GoodsMovementType = '101' or GoodsMovementType = '102' )
group by PurchaseOrder, PurchaseOrderItem
  3. Create ZC_PRPOSchedLineSummary — aggregates schedule line data from I_PurOrdScheduleLineAPI01. Takes MIN and MAX of delivery dates to capture the delivery window; uses MAX (latest date) for the DIFOT on-time comparison.
  4. Create ZC_PRPODIFOT — the core calculation view. Joins the two aggregation views with I_PurchaseOrderItemAPI01, I_PurchaseOrderAPI01, and I_POSupplierConfirmationAPI01 (left outer). Calculates all six DIFOT fields using CASE expressions and DATS_DAYS_BETWEEN for date variance.
  5. Create ZC_PRPOGRHistoryLines — individual (non-aggregated) GR lines from I_PurchaseOrderHistoryAPI01. Same filter scope as the summary view. This is used purely as a navigation target for the Object Page GR Lines tab.
    Must be activated before ZC_PRPODIFOT_C.
  6. Create ZC_PRPODIFOTStatusVH — fixed-value list for the DIFOT Status filter dropdown. Uses an inline CDS VALUES clause — no database table needed.
    Must be activated before ZC_PRPODIFOT_C.
  7. Create ZC_PRPODIFOTFailReasonVH — fixed-value list for the Failure Reason filter dropdown (four values: blank, SHORT, LATE, SHORT AND LATE).
    Must be activated before ZC_PRPODIFOT_C.
  8. Create ZC_PRPODIFOT_C — the consumption view. Selects from ZC_PRPODIFOT, adds all @UI.* annotations for the List Report and Object Page, declares the _GRLines navigation association, wires up all value helps, and computes the DIFOTCriticality virtual field (1=red, 2=orange, 3=green) that drives the colour coding on DIFOTStatus.

Phase 2 — Create the OData V4 service in ADT

  1. Create Service Definition ZC_PRPODIFOT_SRV — exposes ZC_PRPODIFOT_C as PODIFOTItem and ZC_PRPOGRHistoryLines as POGRHistoryLine. 
    define service ZC_PRPODIFOT_SRV {
  expose ZC_PRPODIFOT_C        as PODIFOTItem;
  expose ZC_PRPOGRHistoryLines as POGRHistoryLine;
}
  2. Create Service Binding ZC_PRPODIFOT_UI_V4 — binding type OData V4 - UI. After activating, click Publish Local Service Endpoint in the binding editor. At this point you can already preview the app directly from ADT.

Phase 3 — Deploy the Fiori app from BAS

  1. Configure a BTP destination pointing to your S/4HANA Cloud system (name: SAP_Business_Application_Studio, properties WebIDEEnabled=true, WebIDESystem=ABAP, WebIDEUsage=odata_abap,dev_abap,abap_cloud).
  2. Generate the Fiori app in BAS using the List Report Page template. In the wizard: connect to your system, select ZC_PRPODIFOT_SRV (OData V4), choose PODIFOTItem as the main entity. Set module name zc_prpodifot, deployment target ABAP (package ZPURCHASING), and FLP semantic object ZPRPODifot / action display.
  3. Preview the generated app in BAS against live data to verify it renders correctly before deploying.
  4. Deploy by running npm run deploy in the BAS terminal. This creates a BSP application (WAPA) and a Launchpad App Descriptor Item (UIAD) in your ABAP repository.

Phase 4 — Grant access and expose the tile

  1. Create an IAM App ZC_PRPODIFOT_TILE in ADT — App Type EXT_UI, link it to the UIAD created by BAS, add the OData V4 service on the Services tab. Publish Locally.
  2. Create a Business Catalog ZC_PRPODIFOT_BC in ADT — add the IAM App to the Apps tab. Publish Locally.
  3. Create a Business Role in the Maintain Business Roles Fiori app. Add the Business Catalog. Grant Read and Value Help access.
  4. Assign the role to your user in the Maintain Business Users app. Log out and back in.
  5. Pin the tile via App Finder on the Launchpad home page. Search for PO DIFOT and pin it.
